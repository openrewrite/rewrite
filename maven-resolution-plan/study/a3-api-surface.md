## 2. Object-identity contracts

1. **`getResolvedDependency(Dependency)` — `==` on `requested`** (`MavenResolutionResult.java:104`). For depth-0 dependencies the algorithm guarantees `ResolvedDependency.requested` is the *same instance* as the element of `ResolvedPom.requestedDependencies` (BFS seeds `DependencyAndDependent(requestedDependency, ...)` with the raw instance and constructs `ResolvedDependency` from `dd.getDependency()`, not the substituted copy — `ResolvedPom.java:1028–1034, 1126–1136`). Those instances in turn ARE the elements of `Pom.dependencies` / `Profile.dependencies` of the declaring pom, because `Pom.resolve` passes `dependencies` directly (`Pom.java:205–218`) and `mergeRequestedDependencies` adds parent/profile instances unchanged (`ResolvedPom.java:613–637`). So a caller may take a `Dependency` from `pom.getRequested().getDependencies()` or `pom.getRequestedDependencies()` and look up its resolution by reference.
2. **`getResolvedManagedDependency(ManagedDependency)` — `==` on `requested` OR `requestedBom`** (`MavenResolutionResult.java:177`). `Resolver.mergeDependencyManagement` stores the `Defined` instance as `requested` (`ResolvedPom.java:975`) and the `Imported` instance as `requestedBom` on every entry flattened out of that BOM (`ResolvedPom.java:941–943`) — again the same instances found in `Pom.dependencyManagement`.
3. **Resolved-graph sharing**: the flat per-scope lists in `MavenResolutionResult.dependencies` and the nested `ResolvedDependency.dependencies` children are the *same objects*; parents' child lists are built by mutating `includedBy.getDependencies().add(resolved)` during resolution (`ResolvedPom.java:1143–1151`). `@JsonIdentityInfo` on `ResolvedDependency` preserves this sharing through serialization.
4. **Cross-document marker graph**: `MavenParser.parseInputs` stitches modules/parents by `unsafeSetParent`/`unsafeSetModules` (`MavenParser.java:168–175`) — a module's marker holds a reference to the parent document's *marker instance* (cyclic). `UpdateMavenModel` rebuilds this graph functionally via `withModules(... updateResult(module) ...)` (`UpdateMavenModel.java:203–214`).
5. **`ResolvedPom.resolve()` returns `this`** when nothing effectively changed (`ResolvedPom.java:191, 278`) — identity is the "no change" signal.
6. **`MavenVisitor.getResolutionResult()`** caches the marker per visitor instance and hard-fails if one visitor instance touches two documents (`MavenVisitor.java:80–99`).
7. **Post-serialization caveat (existing behavior, not a regression to avoid)**: `Dependency` has no `@JsonIdentityInfo`, so after an LST round-trip `ResolvedDependency.requested` is *not* `==` to the corresponding `Pom.dependencies` element; contract 1 is only reliable within the JVM run that produced the resolution (or after `UpdateMavenModel` re-resolves). `ManagedDependency` *does* carry `@ref`, so contract 2 survives round-trips of a single document. A new backend must not make this worse (i.e., must preserve in-run instance threading exactly).
8. In-repo callers of 1/2 are absent (the methods' only production references are their definitions) — but they are public API consumed by downstream recipe modules (rewrite-spring, rewrite-java-dependencies, Moderne), so both must keep working. `dm.getRequestedBom() == null` is also used as a "not from a BOM" test (`UpgradeDependencyVersion.java:366`).

## 3. Serialization compatibility

- **Graph shape**: `@JsonIdentityInfo(generator = IntSequenceGenerator, property = "@ref")` on `MavenResolutionResult`, `ResolvedPom`, `ResolvedDependency`, `ManagedDependency`, `MavenRepository`, `GroupArtifactVersion`, `ResolvedGroupArtifactVersion`. First occurrence written in full with `"@ref": n`; repeats as bare int. Any change to which types carry identity info changes the wire format.
- **Old-payload bridging**: `rewrite-core/src/main/java/org/openrewrite/internal/BackwardCompatibleObjectIdModule.java` — non-Marker types with object ids get a deserializer that accepts (a) bare int back-refs, (b) objects with `@ref`, (c) **objects without `@ref`** (pre-annotation LSTs). Markers are excluded because `MavenResolutionResult` is built through its **no-arg `@JsonCreator`** which tolerates a missing id (lines 61–66). ⇒ MUST keep: private no-arg creator on the marker; property-based creators on `@Value` types; do not convert the marker to a creator-with-properties.
- **Polymorphism**: `ManagedDependency` uses `@JsonTypeInfo(Id.CLASS, "@c")` — serialized class names are part of the format; `Defined`/`Imported` cannot move packages.
- **WRITE_ONLY fields**: `MavenResolutionResult.managedReference` (accepted on read, never written by Jackson; written by external reference-managed serializers); `MavenRepository.username`/`password` (credentials never persisted — re-injected from `MavenSettings` at run time).
- **Null-tolerant backward compat** (LSTs serialized years ago must deserialize AND behave):
  - `MavenResolutionResult.getActiveProfiles()` / `getUserProperties()` null → empty (`MavenResolutionResult.java:82–92`); `UpdateMavenModel.java:56–59` re-checks null before use.
  - `ResolvedPom.subprojects` `@Nullable` "on older LSTs this field is not yet present" (`ResolvedPom.java:138–141`); `Pom.subprojects` likewise; `MavenResolutionResult.isMultiModulePom` treats null as "maybe multi-module".
  - `ResolvedPom` ctor: `initialRepositories == null` → keep default, then first `resolve()` back-fills `initialRepositories = repositories` (`ResolvedPom.java:89–91, 433–439`).
  - `dependencyManagementSorted` defaults false so **older unsorted payloads are sorted on first read**; `@JsonPropertyOrder` guarantees the flag is written *after* the (sorted) list (`ResolvedPom.java:49–56, 108–112, 143–156`). Binary-search lookup correctness depends on this exact comparator.
  - `Dependency.attributes` nullable with null-safe getter (`Dependency.java:53–60`).
  - `ResolvedDependency.effectiveExclusions` nullable with null-safe getter (`ResolvedDependency.java:74–80`).
- **Enum wire names**: `Scope` constants serialize as the `dependencies` map keys; `License.Type` also serialized. Renaming any constant breaks old LSTs.
- **Persisted POM caches** (distinct from LSTs): `Pom.getModelVersion()` (=3) gates `RocksdbMavenPomCache` (`cache/RocksdbMavenPomCache.java:249–284`). Model changes to `Pom` or referenced types require a bump (cheap: cache rebuilds). LST compat, by contrast, cannot be "bumped".
- **What this constrains**: new fields must be nullable or defaulted with null-tolerant getters; existing fields cannot change type or name; the *values* stored must remain semantically what old readers expect (e.g. `ResolvedPom.repositories` post-interpolation, `Pom.properties` verbatim). Old readers (older rewrite versions reading LSTs produced by the new backend) additionally require not *adding* `@JsonIdentityInfo` to currently-identity-free types like `Dependency`.

## 5. The re-resolution loop (recipe mutates XML → model recomputed)

- **Trigger**: recipes call `MavenVisitor.maybeUpdateModel()` which schedules a single `UpdateMavenModel` after-visit (dedup by scanning `getAfterVisit()`, `MavenVisitor.java:240–247`). 31 production files in rewrite-maven use it (AddDependencyVisitor, AddManagedDependency, AddParentPom, AddPropertyVisitor, AddRepository, ChangeDependencyGroupIdAndArtifactId, ChangeExclusion, ChangeManagedDependencyGroupIdAndArtifactId, ChangeParentPom, ChangePluginGroupIdAndArtifactId, ChangeProjectVersion, ChangePropertyValue, ExcludeDependency, ManageDependencies, ManagedToRuntimeDependencies, RemoveDependency, RemoveDuplicateDependencies, RemoveManagedDependency, RemoveProperty, RemoveRedundant*, RenamePropertyKey, UpgradeDependencyVersion, cleanup/*, etc.).
- **Inputs re-read from the mutated XML** (`UpdateMavenModel.visitDocument`, `UpdateMavenModel.java:38–155`): `<properties>` (the requested `Pom.properties` map is **cleared and repopulated in place** — lines 45–53 — then overlaid with marker `userProperties`, lines 56–59), `<parent>` (with/without, lines 61–71), `<dependencies>` (rebuilt as `Dependency.builder()` with scope defaulting `"compile"`, lines 73–95), `<dependencyManagement><dependencies>` (`Imported` when `scope=import`, else `Defined`, lines 97–124), `<repositories>` (7-arg `MavenRepository`, lines 126–139). **Not re-read**: plugins, pluginManagement, profiles, licenses, modules — a recipe changing only `<build><plugins>` will not see the model change through this path.
- **Recompute** (`updateResult`, lines 197–218): build `MavenPomDownloader(projectPoms, ctx, marker.mavenSettings, marker.activeProfiles)` where `projectPoms = resolutionResult.getProjectPoms()` **plus the freshly mutated requested pom keyed by sourcePath** (lines 142–146); call `resolutionResult.getPom().resolve(ctx, downloader)` (full property/DM/plugin re-merge against parents, honoring `initialRepositories`); recursively `updateResult` each module **best-effort** (failure keeps the module's previous resolution — lines 205–213); then `resolveDependencies(downloader, ctx)` re-resolves the 4 scopes.
- **Reactor contract**: `MavenPomDownloader` consults `projectPoms` (by path, by GAV, property-merged GAV, and relative-path match) **before any repository** (`internal/MavenPomDownloader.java:76–77, 574–586` and `ResolvedPom.java:580–594` for `${...}`-versioned parents) — so in-repo parents/siblings resolve without network even while coordinates are mid-mutation.
- **Marker replacement**: `document.withMarkers(markers.computeByType(old, (o, ignored) -> updated))` (lines 150–151) — same marker `id` retained (no `withId`), object replaced. Also marks `JavaSourceSet` dirty for this document and every module name recursively (ambiguity-sensitive Java recipes re-check classpaths, lines 157–179).
- **Direct (non-UpdateMavenModel) re-resolution**: `ChangeParentPom`/`UpgradeDependencyVersion.retainVersions` and `AddManagedDependency` construct downloaders directly and call `Pom.resolve`/`ResolvedPom.getManagedVersion` on downloaded BOMs; gradle's `GradleDependencyConfiguration.resolve()` does the same headlessly (§4.2).

## 6. `MavenParser` API surface (`MavenParser.java`)

- **Builder** (lines 190–229): `MavenParser.builder()`, `.activeProfiles(String...)` (accumulating set), `.property(key, value)` (injected into every parsed pom's *requested* properties — implemented by mutating the pom's property map, lines 78–81), `.skipDependencyResolution(boolean)` (marker attached with empty `dependencies` map, lines 120–122), inherits `Parser.Builder(Xml.Document.class)`; `getDslName() = "maven"`.

## 7. Compatibility checklist for the new backend

### MUST (behavioral / API — breaking any of these breaks recipes or serialized LSTs)

**Types & serialization**
1. Keep every class/field/getter in §1 source- and binary-compatible: names, nullability, `@With` withers, builders, public constructors (incl. `ResolvedDependency`'s public `@AllArgsConstructor` and `MavenRepository`'s 9-arg ctor — external constructors are used by rewrite-gradle-tooling-model).
2. Preserve Jackson wire format exactly: `@JsonIdentityInfo("@ref")` on the same seven types (add to none, remove from none); `ManagedDependency`'s `@JsonTypeInfo(Id.CLASS,"@c")` with unchanged FQCNs; `WRITE_ONLY` on `managedReference` and repository credentials; `ResolvedPom` `@JsonPropertyOrder` + lazy-sort `getDependencyManagement()`; marker's no-arg `@JsonCreator`.
3. Preserve null-tolerant getters and old-LST defaults: `activeProfiles`, `userProperties`, `subprojects`, `initialRepositories`, `attributes`, `effectiveExclusions`, `dependencyManagementSorted=false` re-sort path.
4. Keep `Scope` constant names and **ordinal order**; keep `License.Type` names.
5. Deserialized-years-ago LSTs must load and answer the same queries (glob finds, managed lookups, property interpolation) — i.e. read paths must not start requiring data that old LSTs lack.

**Instance threading & mutability**
6. `ResolvedDependency.requested` must be the *same instance* as the `Dependency` in `ResolvedPom.requestedDependencies` / declaring `Pom.dependencies` (depth 0); `ResolvedManagedDependency.requested`/`requestedBom` must be the same instances as `Pom.dependencyManagement` entries — `getResolvedDependency`/`getResolvedManagedDependency` use `==` (`MavenResolutionResult.java:104,177`).
7. The resolved graph must reference-share: flat scope lists and nested `dependencies` children are the same objects (serialized via `@ref`).
8. `Pom.properties` must remain a mutable `Map` that resolution reads live — `MavenParser` (lines 78–81) and `UpdateMavenModel` (lines 45–59) mutate it in place before re-resolution.
9. Keep `unsafeSetParent`/`unsafeSetModules`/`unsafeSetManagedReference` and the cyclic marker graph semantics; keep `ResolvedPom.resolve()` returning `this` on effective no-change.

**Resolution semantics observable through the model**
10. `dependencies` map contains exactly `{Compile, Runtime, Test, Provided}` in that insertion order; same dependency may appear in several scopes; per-scope lists ordered as today (BFS by depth, declaration order within depth).
11. Conflict rules reproduced: nearest-wins with "direct duplicates: last declaration wins" / "same-depth transitives: first wins"; transitive type filter (`jar|ejb|pom|zip|bom|tgz`); optional pruning; exclusion propagation with `effectiveExclusions` attributed to the declaring ancestor; managed-scope precedence rule between containing pom and project pom; depth-0 versionless dependency → `MavenDownloadingException`; property-unresolvable GAV → `MavenDownloadingException` not a crash.
12. Requested-vs-effective duality: `Pom` stays *verbatim* (placeholders unresolved, groupId/version inherited at `toPom`, profiles reversed, licenses classified); `ResolvedPom` carries the merged/interpolated view; `getValue`/`getProperty` semantics incl. `System.getProperty` override, `project.*` pseudo-properties, null-property→`""`, circular-version sentinel.
13. Reactor-first download: project poms (path/GAV/property-merged/relative-path, `${...}`-versioned parents) resolve before any remote repo; `.m2`-relative parents excluded from `getProjectPoms`.
14. BOM flattening with `requestedBom`/`bomGav` back-links; `LATEST`/`RELEASE` managed versions resolved via metadata; parent-cycle detection; obsolete-pom refusal (`obsoletePomVersion`).
15. Errors: `MavenDownloadingException`/`MavenDownloadingExceptions` types, `e.warn(doc)` markup, parse-not-failed-on-resolution-failure, per-GA exception dedup across scopes (`MavenResolutionResult.resolveDependencies`).
16. Fire `ResolutionEventListener` events at equivalent points (downloadError with attempted URIs is load-bearing for host diagnostics).

**Entry points**
17. `MavenParser.Builder` options (`activeProfiles`, `property`, `skipDependencyResolution`), accept rules, module/parent stitching, sanitized settings on marker.
18. `MavenVisitor` helper methods (tag classification, find*, downloadMetadata against `getPom().getRepositories()`/`getPluginRepositories()`, `maybeUpdateModel`) behave identically.
19. `UpdateMavenModel` contract: re-reads exactly properties/parent/dependencies/dependencyManagement/repositories from XML, re-resolves against the reactor incl. modules best-effort, replaces marker via `computeByType`, marks JavaSourceSet dirty.
20. Headless engine reuse: `Pom.builder()` + `pom.resolve(profiles, downloader, ctx)` + `resolvedPom.resolveDependencies(scope, downloader, ctx)` / `resolveDirectDependencies` must work for synthetic in-memory poms with no XML (rewrite-gradle depends on it), and `MavenPomDownloader`'s public constructors (`(Map<Path,Pom>, ctx)`, `(Map, ctx, settings, activeProfiles)`, `(ctx)`) must remain (they are called from rewrite-gradle and recipes even though the package is `internal`).
21. Pluggable caches preserved: `MavenPomCache` (incl. `getResolvedDependencyPom`/`putResolvedDependencyPom` keyed by `ResolvedGroupArtifactVersion`, negative repo caching), `MavenArtifactCache`, all `MavenExecutionContextView` knobs (mirrors, credentials, local repo, add-central/local, pinned snapshots, settings merge via `effectiveSettings`). Bump `Pom.getModelVersion()` if the cached `Pom`/`ResolvedPom` shape changes.

### SHOULD (performance / quality goals)
1. **Raw speed** stays first-class: keep interprocess-parallel-friendly cache design (the original motivations: no `~/.m2` lock contention, pluggable caches); preserve negative caching of dead repositories (`repositoryAccessFailedPreviously` event, `getUnreachableEndpoints`) so dead repos are not re-hit — this was motivation (2) for the custom algorithm.
2. Keep `InMemoryMavenPomCache.deduplicate()`-style memory discipline and `resolveDirectDependencies`'s avoid-transitive-download fast path; keep `ResolvedPom.resolve()`'s no-change short-circuit so `UpdateMavenModel` stays cheap when recipes make non-model edits.
3. Determinism: resolution output should depend only on inputs (pom bytes, settings, ctx) — today's `System.getProperty` fallback in `getProperty` (`ResolvedPom.java:323,372`) and `System.getenv` profile activation (`ProfileActivation.java:85–92`) are existing leaks; match Maven's actual `-D`/property-activation semantics rather than replicating the env-var quirk, but document any behavior change (property-activated profiles currently activate off env vars — arguably a bug worth fixing *with* the switch).
4. Readability: keep the requested/effective split (`Pom`/`ResolvedPom`) as the seam between "Maven API adapter" and the frozen public model; isolate all Maven-API interop behind the downloader/resolver so the tree package remains dependency-free (`tree` types must not leak Maven/aether classes — they are serialized into LSTs).
5. Preserve version-comparison behavior (`tree/Version` is the aether GenericVersion port; `VersionRequirement` range semantics) or delegate to the real aether classes with equivalent results.
6. Keep `MavenMetadata` XML parsing lenient (null `versions` normalization, `MavenMetadata.parse(byte[])`, `deriveMetadataIfMissing` on repos).
7. Test using the existing corpus: `ResolvedPomTest`, `RawPomTest`, `MavenParserTest`, `UpdateMavenModel`-dependent recipe tests, plus a serialized-LST fixture round-trip (old-format payload without `@ref`) to lock §3.

### Known divergences safe to fix (not compat-bound, but announce)
- Profile property activation via env vars instead of system properties (`ProfileActivation.java:85–92`).
- Effective-property leak into `Pom.properties` via map aliasing (`Pom.java:205–218` + `ResolvedPom.mergeProperties`).
- `UpdateMavenModel` not re-reading `<build><plugins>`/`<profiles>` — today's behavior; matching it is MUST for now, extending it is an improvement.
