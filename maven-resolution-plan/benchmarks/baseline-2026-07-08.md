# Large-reactor benchmark baseline — 2026-07-08

Companion to `baseline-2026-07-08.json` (machine-readable; environment + methodology + per-tier numbers).
Harness: `maven-resolution-plan/spike/benchmark/`, parameterized this session (`-Pcorpus`, `-PmaxModules`,
`-PleafModule`, `-PskipCold`, `-PlocalRepo`). Corpora: apache/maven@maven-3.9.16 (15 modules, full) and
apache/camel@camel-4.4.0 (548 non-src reactor poms) at graduated tiers 50/150/400/full-548.

**Tier selection rule** (`Corpus.selectSubtree`, reproducible): all 47 pom-packaging modules
(parents/BOMs/aggregators — the scaffolding that exercises the shared-parent ModelCache path) are always included;
remaining artifact modules fill the cap in path-sorted order. Caps are strict supersets (50 ⊂ 150 ⊂ 400 ⊂ 548).
Cold runs skipped for camel (warm scaling is the question); warm = median of 7 after 2 warm-ups; loop = same leaf
module ×50, pinned identically for both engines per tier.

## The ModelCache clone-scaling verdict

Maven 3.9's `DefaultModelBuilder` deep-clones on every cache hit (`RAW` → `Model.clone()`, `IMPORT` →
`DependencyManagement.clone()`). Measured per warm full-reactor build (NEW engine):

| corpus | modules | RAW clones | IMPORT clones | total | clones/module |
|---|---:|---:|---:|---:|---:|
| maven | 15 | 44 | 15 | 59 | 3.9 |
| camel | 50 | 181 | 193 | 374 | 7.5 |
| camel | 150 | 632 | 602 | 1,234 | 8.2 |
| camel | 400 | 1,684 | 1,613 | 3,297 | 8.2 |
| camel | 548 | 2,353 | 2,217 | 4,570 | 8.3 |

**Clones grow linearly in module count — no superlinear blowup, no cliff.** Camel's marginal slope is flat at
~8.3–8.6 clones/module across 50→548 (maven's coefficient is 3.9/module — the coefficient is parent-chain depth ×
import usage, corpus-dependent, but scaling in module count is linear in both corpora).

**Clone cost is bounded and small.** Warm N1 grew 1,036 → 2,167 ms while clones grew 374 → 4,570 over 50 → 548
modules. Even attributing *all* marginal warm growth to clones gives ≤0.27 ms/clone; the true share is far lower
(collection dominates the marginal work). At 548 modules the full warm build is 2.17 s with 4,570 clones inside it.

**Mitigation verdict (DESIGN.md §6): the custom clone-free ModelCache is NOT necessary at any measured size (≤548
modules).** Linear extrapolation says a ~3,000-module reactor would pay ~25k clones/warm build — even at the
pessimistic 0.27 ms bound that is ~7 s, and realistically several times less. Revisit only if reactors well beyond
~1–2k modules become a target, or if Phase-4 profiling shows the model phase (not collection) dominating.

† OLD-548 was skipped (each OLD tier pays a 20–40 min unmeasured serial network warm-up; the OLD curve already has
4 points). OLD's warm cost is itself cleanly linear (~8.0–8.4 ms/module marginal), so OLD-548 ≈ 5.2 s; ratios marked
approximate. All other rows compare identical module sets (modules= recorded in both engines' RESULT lines; same
pinned loop leaf per tier, asserted at JSON-assembly time).

**The bigger warm-path finding is N1-vs-N4, not clones.** On the BOM-heavy camel reactor, four per-scope collects
(N4 — rewrite's current four-scope model) are *slower than the OLD engine* at 50–150 modules (0.55–0.75×) and only
barely ahead at 400 (1.31×). The single-verbose-collect + in-memory scope mediation design (N1) stays ahead of OLD
at every tier and scales ~3.5× shallower (NEW-N1 marginal ~2.3 ms/module vs OLD ~8.2 ms/module). On maven-15 the
N4 penalty was a tolerable 1.65–1.83×; on camel it is ~2× of a much larger base. **Adopt N1 scope mediation for the
production engine; N4 would forfeit the warm-path win exactly where it matters (large reactors).**

The re-resolution loop (UpdateMavenModel-style, one leaf) is size-independent for NEW (~2.1 ms at every tier) and
2–4.6× faster than OLD everywhere.

## Measurements re-taken (and why)

1. **camel OLD-150 — re-measured.** The first measurement (warm 4,756 ms, loop 9.9 ms) was invalid: that run alone
   pointed its local repository at the resolver-populated repo (a warm-up shortcut experiment); rewrite-maven treats
   the local repo as a `file://` repository, so every warm iteration paid thousands of local file reads the other
   OLD tiers did not (the persistent 2× inflation of the ~5 ms loop metric on the same leaf was the fingerprint),
   compounded by proximity to a machine-suspend window. Re-run under the standard empty-local-repo methodology:
   warm 2,016.4 ms — restoring OLD monotonicity (1,177 → 2,016 → 4,016 ms).
2. **camel OLD-400 — killed and re-run.** The machine was suspended (laptop lid) during its unmeasured network
   warm-up; the process was killed and re-run fresh to completion (final: tight 3,958–4,111 ms spread).
3. **Suspend audit over all other tiers:** every recorded warm spread has max/min ≤ 1.17 (a mid-measurement suspend
   shows as an extreme outlier); no other tier was affected.

## Caveats (honest numbers over favorable)

- NEW remains the *unadapted* pipeline (no LST mapping); NEW numbers are a floor, as in `spike/benchmark/RESULTS.md`.
- camel OLD tiers 150/400 each had 2 parse failures (poms yielding no resolution result; NEW built all models at
  every tier, modelFail=0/collectFail=0).
- camel NEW warm iterations include ~6 residual network probes/iter (fresh session re-probes a handful of
  never-resolvable poms); this slightly inflates NEW camel numbers — the unfavorable direction for NEW.
- graphNodes are not engine-comparable by design (OLD: winner-only ×4 scopes; NEW-N1: verbose single scope; NEW-N4:
  verbose ×4 overlapping scopes). They confirm work scales with tier size within each engine.
- camel tier-50's loop leaf is `camel-debezium-common` (deterministic fallback; that tier's fill-leaves are 3
  archetypes) while tiers ≥150 pin `camel-activemq` — loop values are engine-comparable within a tier, not across
  the 50→150 boundary.
- Environment: Mac16,7 (14 cores, 24 GB), Temurin 17.0.6, `-Xms2g -Xmx2g`, separate JVM per engine, Gradle 9.5.1;
  rewrite-maven 8.86.1 vs maven-resolver 2.0.20 + maven-model-builder 3.9.16.
