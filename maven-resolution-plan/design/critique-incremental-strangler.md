**Verdict: SOUND-WITH-CHANGES** — the architecture (frozen tree API, policy-wrappers-around-Maven, ledger-gated strangler) is right, but three findings are load-bearing enough that the design as written would regress its own stated goals.

**1. The warm-cache story is broken for persistent POM caches (severe).** §5.1 makes LRM *files* the model builder's XML source, and a raw-`Pom` cache hit skips aether entirely ("warm hit = no aether") — so nothing lands in the LRM. On the flagship server-side config (Moderne: warm RocksDB/Composite pom cache, cold/ephemeral LRM), Phase 2 has no XML to build from, and `Pom` cannot reconstruct it (RawPom drops distributionManagement, relocations — a1 §1.1). Either every pom re-downloads (perf/pluggability break) or you rebuild lossy XML (divergence). The design defers this as open question #3; it is not open, it is the primary hot path. **Fix:** adopt a pom-bytes sub-cache (default methods on `MavenPomCache`, keyed like `getPom`, materialized into an LRM scratch dir on demand) in Phase 1 — the full-delegation design's §6.1 already has this shape.

**2. The `getResolvedDependencyPom` L1 quietly rebuilds a parallel semantic engine.** On a hit, `CachingDescriptorReader` must synthesize an `ArtifactDescriptorResult` (interpolated deps, managed deps, repos) from a rewrite `ResolvedPom`, while misses serve Maven-built descriptors. That is two producers of descriptor semantics — the exact divergence class the project exists to kill — now warm-vs-cold instead of rewrite-vs-Maven, plus the retained GAV-only key contamination (a1 §4.19) now feeds Maven's collector. **Fix:** cache the effective `Model`/descriptor result itself under this key (modelVersion bump covers it; RocksDB never persisted this sub-cache anyway, a2 §2.3), or benchmark ModelCache-only first and add the L1 only if gates fail.

**3. Stale-model invalidation under `UpdateMavenModel` is unaddressed.** The session `RepositoryCache` (ModelCache RAW/IMPORT entries, DataPool descriptor/subtree pools) is deliberately held "across UpdateMavenModel re-resolutions" (§6.3), but those caches key by GAV — a recipe that mutates a parent pom *without changing its GAV* (add a managed dependency, then child re-resolves) gets the stale cached model. Today this is safe because `projectPoms` short-circuits before any cache. **Fix:** epoch the reactor into the cache key (workspace-reader generation counter) or evict project-pom GAVs on marker replacement; add a mutate-parent-then-re-resolve test to Phase 2's green criteria.

**4. "Nothing semantic stays custom" (Phase 3) is overstated.** Per-scope pure-list packaging (`Scope.transitiveOf`, a1 §3.7), the empirically-tuned scope-widening guard (a4 #22), root-scope=Compile seeding, and `effectiveExclusions` attribution (#8117 — which requires knowing what the exclusion selector *pruned*, invisible even in verbose graphs) all remain rewrite semantics living in `DependencyGraphMapper` and custom selectors. That's defensible, but the design must own it: specify the four selector configurations and the widening-guard location, and file these as KEEP_REWRITE ledger entries rather than claiming elimination.

**5. Parser/ctx-injected properties compat gap.** `MavenParser.builder().property()` and `UpdateMavenModel`'s userProperties overlay mutate `Pom.properties` in place today (a3 MUST #8), so they're visible through `ResolvedPom.getProperties()/getValue()`. §4.2 builds the properties map from *Maven's raw lineage models*, which never saw those injections; routing them only as Maven user-properties fixes interpolation but regresses `getValue()`/map contents (`projectVersionPropertyOverriddenByBuilderProperty`, the maven.config tests, a4 #12/#34). **Fix:** `EffectivePomMapper` must also overlay injected properties into the merged map; state it explicitly.

**6. CACHE_ALL persists transfer errors across processes** — `.lastUpdated` markers suppress retries until update-policy expiry (default daily, a5 §2.2). Today a blip is retried next process; RocksDB's "never persist negatives" precedent says hosts want that. §5.1/§11.10 frame this as "strictly stronger"; it's a behavior change that can poison CI for hours. **Fix:** persist CACHE_NOT_FOUND only; keep transfer-error memory session-scoped (sessionState dedupe already covers within-run) or expose the interval as a ctx knob with a ledger entry.

**7. The oracle is lossy exactly where the API is strict.** Normalizing to sorted sets discards per-scope ordering (a3 MUST #10), nested graph shape (`ResolvedDependency.dependencies` edges recipes traverse), instance threading, and event streams. Identity tests exist (§8.4) but not at corpus scale. **Fix:** diff ordered lists + a canonical nested-graph serialization, and run identity probes and an event-multiset comparison inside `ResolutionShadowComparator` (the competing design's §9 does this).

**8. The 1.0–1.5× warm estimate has no measurement basis** — no study contains a benchmark; 4 verbose collects + classic `ConflictResolver` (O(N²) worst case, a5 §4.3) per module on a 300-module reactor is asserted, not sized. The gate structure is right; the number isn't earned. **Fix:** run the big-graph benchmark as a week-1 unshaded spike before committing 39 pw (vs the rival's 24 — justify the delta), with named fallbacks (path resolver, lineage memoization).

**9. Minor:** shipping shaded Apache HTTP is dead weight and an I/O-bypass hazard when `HttpSenderTransporter` is the sole transporter (a2 §5 hard requirement) — drop transport-apache, keep transport-file.

**What it gets right that the rival likely misses:** the divergence *ledger* as a reviewed acceptance contract (not just a triage report); benchmarks as per-phase ship-blockers; the candid §11 non-preservables (`clear()`, event ordering, read-time interpolation skew); applying mirrors at request-repo-list assembly rather than the session `MirrorSelector` (which only touches descriptor-discovered repos — a5 §2, an easy silent bug); keeping `ResolvedPom.properties` *raw* via lineage merge instead of Maven's interpolated map; preserving `MavenPomDownloader`'s public constructors (a3 MUST #20); and grounding the mvn3-on-Java-8 choice in parity-with-what-users-run, with a genuine `MavenEngine` escape hatch to Maven 4.
