**Verdict: SOUND-WITH-CHANGES** — the stack choice, seam placement, and oracle wiring are right; but three findings are correctness breaks that would ship regressions, and two study-cited claims don't survive scrutiny.

**1. Cross-scope instance sharing breaks per-scope graph shape (API break).** §5.3 says one `ResolvedDependency` per winner, shared across the four scope lists. Today each scope is resolved independently: the Compile-list instance of a dep has *compile-classpath-only* children and a scope-specific depth (a1 §3.6: children enqueued only if `d2Scope.isInClasspathOf(dd.getScope())`; links built per scope-resolution). A shared instance must have one `dependencies` list — so `findDependencies` inside the Compile list would now traverse runtime/test-reachable children. That's an observable recipe-facing regression independent of the admitted single-mediation version shifts (§10.6). Fix: from the single mediated graph, project *per-scope* `ResolvedDependency` instances with scope-filtered children (share only within a scope, as contract #3 actually requires); spec this derivation explicitly.

**2. The warm-path memo serves stale results for mutated reactor POMs.** §7 step 1 keys the effective-model memo on `(gav, activeProfiles, repos)`. `UpdateMavenModel` mutates `Pom.properties`/dependencies *in place* without changing the GAV (a3 §5), so the hottest recipe loop (31 recipes → `maybeUpdateModel`) hits a stale memo and `resolve()` falsely returns `this` — edits silently stop propagating. Today's `getResolvedDependencyPom` only ever caches *remote* POMs. Fix: bypass the memo for workspace/reactor GAVs (or key on a content fingerprint of the pom + its reactor ancestry).

**3. The deep-clone mitigation doesn't cover reactors — the 1.2× target is unsubstantiated where it matters most.** §7's defense of `ModelCacheTag`'s clone-per-hit is "the memo sits above ModelBuilder, clones paid once per distinct (gav, context)". But in a 300-module reactor every module *is* a distinct GAV; the clones occur on the RAW/IMPORT cache hits for each module's parent chain and BOMs, per module — the memo can't amortize them. Today's passes are map lookups over shared `Pom`s. Fix: Phase-0 JMH must include a deep-parent-chain reactor benchmark specifically; budget a `ModelCache` that avoids `ModelCacheTag` cloning (own cache impl storing pre-cloned immutable snapshots, clone-on-read once) as a planned mitigation, not a discovery.

**4. Profile-plugins "fix" contradicts the UpdateMavenModel contract.** §5.2 celebrates honoring profile `<build><plugins>` (sourced from real XML), but §6.5 re-resolves mutated poms via `PomToModelConverter`, which can only carry what `Pom` carries — and `RawPom` never deserializes profile builds (a1 §1.1). Result: plugins appear on first parse, vanish after the first `maybeUpdateModel`. Fix: pick one — add profile builds to `RawPom`/`Pom` (modelVersion bumps anyway), or defer the fix and convert from `Pom` always.

**5. Engine-internal metadata bypasses the pluggable cache, pinning, and the negative-caching stance.** Ranges/LATEST/snapshots inside collect go through Maven's `VersionResolver`/`MetadataResolver` (per-repo files + touch files), not `MavenPomCache.getMavenMetadata`, not rewrite's cross-repo merge (a2 §1.3), and the facade's snapshot-pinning "pre-pass" never sees *transitive* snapshot deps resolved inside aether. Worse, `.lastUpdated` error markers persist not-founds across processes — contradicting the RocksDB "never persist negatives" stance §6.1 claims to preserve. Fix: decorate `createVersionResolver`/metadata resolution to route through `MetadataDownloader`+`MavenPomCache` and consult `pinnedSnapshotVersions`; set update-policy/session config (or per-run scratch dirs) so no negative outlives the run.

**6. Identity-threading joins are fragile exactly where recipes need them.** Transitive `requested` matched by GACT fails when G/A/classifier are property references (`${project.groupId}` is the *default* for siblings, a1 §3.6); `Defined` DM matched "positionally within the model" breaks under `ModelNormalizer.mergeDuplicates` (a6 §1.4 de-dupes declarations) and profile injection. Fix: join by `InputLocation` line/column everywhere (location tracking is already on), and extend the comparator's identity probes to transitive nodes and DM entries over property-heavy fixtures.

**7. The parent-cycle shim is probably dead on arrival.** `DefaultModelBuilder` has its own FATAL `parentIds` cycle check (a6 §1.4); a stub served *with the revisited GAV* still trips it. The catch-partial-result fallback exists only for the root. Fix: spike this first in Phase 2 (a4 #8/#16/#83 fixtures as the entry gate); likely shape is a stub with a synthesized non-colliding model id.

**8. Oracle corpus is biased toward the easy half.** Top-500 Central POMs exercise none of catalog sections E/G: mirrors, auth+anonymous fallback, snapshots/classifiers, dead repos, HTML-index derivation, gradle-metadata, `file://` semantics. Ground truth is only `dependency:tree`. Fix: add a MockWebServer-backed synthetic corpus keyed to a4's E/G entries, and capture `help:effective-pom` for effective-model ground truth.

**9. "No rewrite code re-implements a merge" is overstated.** The property fold, requestedDependencies merge, repository pipeline, and *two* glob-exclusion implementations (`GlobExclusionSelector` + `ExclusionAttributor`) re-encode precedence semantics. Fix: test the folds against Maven's own effective output; share one glob matcher between selector and attributor.

**10. Minor gap:** on a bytes-cache hit the parsed-`Pom` region is never populated (`putPom` only on transporter success), starving §5.3's instance threading and license lookups on warm runs. Fix: parse-through in `CacheBridge` on bytes hits.

**What it gets right** (no competing design file was present to compare): choosing supplier-mvn3 over a6's Maven-4 recommendation on Java-8 and ecosystem-semantics grounds is correct and well-argued; rejecting host-injected `RepositorySystem` kills the per-host-version divergence class; `Transporter.classify()` as the single bridge into Maven's negative-caching machinery is exactly the a5 judgment; POM-only collect leaving `MavenArtifactCache` untouched is a real insight; `RawPom→Pom` as the frozen wire anchor plus the JUnit double-engine extension turning 5,200 existing tests into a differential oracle is the strongest part of the plan.
